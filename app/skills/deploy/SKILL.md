## Навык: деплой сайтов

Ты деплоишь сайты, размещённые под `/opt/<site>`, выполняя команды на самом
хосте (через `nsenter`). Есть два инструмента:
- `deploy_plan` — **только чтение**: определяет способ деплоя сайта и показывает
  точную команду, ничего не запуская. Безопасен.
- `deploy_run` — **выполняет деплой** на хосте. Опасен, система запросит
  подтверждение пользователя.

### Порядок работы
1. Сначала всегда `deploy_plan <site>` — узнай метод (`script`/`make`/`compose`)
   и команду. Покажи пользователю, что именно будет запущено и в каком каталоге.
2. Затем `deploy_run <site>` — запусти деплой (пользователь подтвердит).
3. Верни фактический результат: `returncode`, ключевые строки `stdout`/`stderr`.
   Не выдумывай вывод.

### Как определяется способ деплоя
- Есть `deployments/deploy.sh` → запускается он (сам делает git pull, миграции,
  статику, nginx). Это основной путь для «умных» сайтов.
- Иначе цель `deploy` в `Makefile` → `make deploy`.
- Иначе есть `docker-compose.yml` → `git pull --ff-only && docker compose up -d --build`.
- Иначе способ не определён — сообщи пользователю, ничего не запускай.

### Правила безопасности
- Деплоить можно только сайты из allowlist (`DEPLOY_ALLOWED`). Остальные —
  инструмент откажет; не пытайся обойти.
- **Никогда не трогай чужую инфраструктуру:** `remnawave`, `nginx`,
  telegram-бота. Даже если пользователь назвал их — уточни, это не наши сайты.
- Если `deploy_run` вернул ненулевой `returncode` — деплой не удался: покажи
  stderr, не заявляй об успехе.
